# 缺口分析:模型驱动的多轮智能体循环(生图由 LLM 决定)

> 目标架构:每个用户回合 = 后端运行一个**智能体循环**(2~n 轮)。每轮模型输出 **JSON 决策**,后端据此执行(纯回复 / 调生图 / 继续下一轮 / 结束),最终聚合为统一 JSON 返回前端。**生图与否完全由模型决定,无用户按钮**。

---

## 一、当前已具备(可复用)

| 能力 | 位置 | 复用方式 |
|---|---|---|
| 多模态润色调用 | `qwen.api.ts: multimodalRefine` | 作为循环中"决策/思考"调用 |
| 图像生成调用 | `qwen.api.ts: generateImages` | 包装为可被模型触发的"动作" |
| 会话上下文 | `session.service` + `buildHistory` | 作为循环的系统级上下文 |
| 图片落盘/静态访问 | `storage.service` / `/api/images` | 生图动作的产出落地 |
| 统一响应信封 | `{success,data,error}` | 最终结果外层沿用 |
| 错误中间件 | `middleware/error.ts` | 捕获循环异常 |

> 结论:零件齐,但**缺一个把它们串成"循环"的控制层**,以及**模型与后端之间的 JSON 契约**。

---

## 二、缺失项(按层)

### 1. 控制层:循环编排器(最关键,目前完全没有)
现状 `chat.routes` 是**线性**的:`润色 → 判模式 → (可能)生图 → 返回`,一次调用一次返回。
需要新增一个 **agent loop runner**:
- 每轮:构造消息 → 调模型 → 解析 JSON 决策 → 执行动作 → 把动作结果回喂下一轮(或终止)。
- **终止条件**:模型返回 `done/final`,或动作不是"生图",或达到 **最大轮数上限**。
- **预算护栏**:单回合最多 N 轮、最多 M 张图(防失控烧钱)。
- **超时/取消**:长循环需要超时与(可选)用户中断。

### 2. 契约层:JSON 决策 Schema(未定义)
模型每轮必须返回稳定 JSON。需规定字段,例如:
```
{
  "thought": "简述判断",
  "action": "reply" | "generate" | "finish",
  "text": "给用户的文字(建议/说明/解释)",
  "imagePrompt": "生图用 Prompt(仅 action=generate)",
  "imageCount": 1,
  "needMore": true | false   // 是否需要再跑一轮
}
```
**未定的关键点**:
- 字段集合最终长什么样(上面只是草案)。
- **如何强制 JSON**:① 提示词约束 + `JSON.parse` + 失败重试;② 千问原生 `response_format: json_object`;③ **function calling / tools**(最稳,把 `generate` 注册为工具)。三者选一。

### 3. 语义层:"是否需要继续生图"的二义性(必须先澄清)
你说"最终 JSON 包含是否需要继续生图",它有两种完全不同的含义,决定整个架构:
- **含义 A(驱动后端自动续跑)**:`needMore` 是循环**继续条件**,模型说 true → 后端自动再跑一轮(n 轮的来源)。循环在 needMore=false 或触顶时结束。
- **含义 B(给前端的提议)**:循环**当轮就结束**,`canContinue` 只是提示前端展示"继续生成"按钮,由用户决定下一步。
- 或 **A+B 混合**:后端按 A 自动跑若干轮,最后再带一个 B 的提示。
> 这是头号待定决策,直接决定循环控制逻辑与前端交互。

### 4. 传输层:多轮的延迟与反馈(当前不足以支撑)
- 现状:单次 `POST /chat/generate` → 阻塞 → 单次返回。多轮 + 生图可能耗时数十秒,阻塞请求体验差且易超时。
- **缺失**:
  - 是否改 **流式(SSE / WebSocket)** 逐轮推送进度(思考中→生图中→完成)?目前完全没有。
  - 若保持阻塞 POST,需明确**单回合总时长上限**与前端 loading 文案分级。

### 5. 状态层:瞬态工作内存 vs 持久会话(未区分)
- 循环内部各轮的 thought/工具结果属于**本次回合的临时 scratchpad**,不应污染持久会话。
- **缺失**:定义"只持久化最终的用户消息 + 最终聚合回复",中间轮次不落盘(或单独临时存储,回合结束丢弃)。

### 6. 执行层:生图作为"动作"的健壮性
- `generateImages` 需改为循环内可**多次调用**的动作,并把**文件名结果回喂模型**(供其下轮判断/收尾)。
- **缺失的失败策略**:生图动作失败时——重试?降级为纯文本?终止并返回已有部分?(需定义)

### 7. 最终响应契约(需重新定义)
最终返回前端的 JSON 需承载:聚合文字 + 全部图片 + 模型决策标志(如建议续生)。当前 `ChatGenerateResponse {sessionId, mode, content, images, count}` **没有**这些。`mode` 字段在模型驱动下已无意义,应移除/替换。

### 8. 前端:结构化结果渲染 + 续生交互(未规划)
- `ChatBubble` 现渲染"一段文字 + 图片数组"。多轮聚合后可能是一段叙述 + 多组图 + 续生提示。
- **缺失**:若采用含义 B/混合,需"继续生成"按钮 UI;以及多轮进度的展示规范。

### 9. 提示词层:智能体人设 + 决策准则(需重写)
- 现提示词是"模式标记"范式,与 JSON 工具范式不兼容。
- **缺失**:
  - 新的**智能体系统提示词**:讲清可用动作、JSON 契约、**何时该设 action=generate 的判定准则**(避免又回到过度生图)。
  - 少量 few-shot 稳定输出。

### 10. 工程保障(缺失)
- **可观测**:每轮 JSON 决策需落日志,便于调试误判。
- **并发/预算**:单会话并发回合限制、全局生图速率限制。
- **可取消**:长循环的中断通道(AbortController / 前端取消按钮)。

---

## 三、需你拍板的 3 个关键决策(决定后续实现)

1. **"继续生图"的语义**(上文第 3 项):A 自动续跑 / B 前端提议 / 混合?——**最优先**。
2. **JSON 强制方式**:提示词+解析重试 / `response_format` / **function calling(推荐最稳)**?
3. **传输方式**:保持阻塞 POST(简单、先跑通)/ 改 SSE 流式(体验好、工作量大)?

---

## 四、一句话总结

> 零件已有,缺的是**"循环编排器 + JSON 决策契约 + 续生语义定义 + 多轮传输/状态/失败策略"**这一整层控制平面;外加与 JSON 工具范式匹配的**新提示词**与**前端结构化渲染**。先定上述 3 个决策,即可据此出详细实现方案。
