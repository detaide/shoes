# shoes — AI 鞋类商品图生成 WebUI · 项目文档

> 一个面向电商场景的 AI 鞋类商品图生成工具。用户用文字描述(可选附参考图),助手判断意图后**生成图片**、**修改图片**或**给出建议/说明**(纯对话)。由 Electron 桌面应用 `autoShoes` 迁移重构而来。

---

## 一、产品定位

- **场景**:电商商品图生产。输入文字或参考图,产出摄影棚级、纯白背景的商品图。
- **底座**:阿里云 DashScope(通义千问多模态润色 + 千问图像生成)。
- **形态**:前后端分离 WebUI(开发期 dev server 代理,生产期可独立部署)。
- **关键差异点**:不是"逢消息必生图",而是**按用户意图决定是否消耗生图额度**——咨询/建议类只回文字。

---

## 二、技术栈

| 层 | 技术 |
|---|---|
| 前端 | Vite + React 19 + TypeScript + React Router |
| 后端 | Node.js + Express + TypeScript |
| 共享类型 | `shared/types`(以 `.d.ts` 声明,前后端共用,编译期擦除、零运行时耦合) |
| 存储 | 服务端磁盘(`data/` 目录)+ `.env` 配置 |
| AI | 千问多模态(润色/意图判断)+ 千问图像(文生图/图生图) |

设计令牌、响应式布局(桌面侧栏 / 移动底栏)由 `useResponsive` 驱动。

---

## 三、核心功能模块

### 1. 智能对话生图(意图驱动)
系统先用多模态模型判断用户意图,落到两种模式之一:

| 模式 | 触发条件 | 行为 |
|---|---|---|
| **生成模式** | 明确动作词(生成/画/做/设计/改成/换成…)+ 具体描述或参考图 | 输出"理由 + 设计说明 + 生图 Prompt",再调用图像模型产图 |
| **对话模式** | 征求评价、求建议、求推荐、问知识/功能、闲聊,或意图不明 | 仅文字回复,**不生图**(省成本);拿不准时反问"需要我帮你生成/修改图片吗?" |

> 对应三大用户场景:① 纯文字→文生图;② 文字+参考图→图生图(改图/新款);③ 咨询建议→纯对话不出图。

生成模式下,回复会**完整说明为何生图与设计思路**(理由 + 设计说明),避免"莫名出图"。

### 2. 会话上下文(Session)
- 每段对话独立为一个 **session**,多轮历史作为上下文喂给润色模型(理解"再来一张""换个颜色"等延续指令)。
- **默认使用最新会话**:每次连接自动加载最近更新的会话。
- 可**新建 / 切换 / 删除**会话(左侧抽屉式会话面板)。
- 会话标题由首条用户消息自动派生。

### 3. 图片复用(图→输入)
气泡内的任何图片(用户上传的参考图、AI 生成的图片)均可在预览弹窗中点击**「用作参考图」**,直接转入输入框,作为下一轮生成的参考——无需重新上传。生成图会自动由文件名转为 base64 注入。

### 4. 图库(Gallery)
- 汇总所有生成图,按时间倒序、响应式网格展示。
- 支持**大图预览、下载、复制到剪贴板、删除**。
- 预览/右键菜单/复制/下载能力与聊天页共用同一弹窗组件。

### 5. 设置(Settings)
- **千问平台**:Base URL、API Key(密钥服务端托管,前端只显示"是否已配置",不回传明文)。
- **模型配置**:多模态模型名、生图模型名。
- **加载策略**:启动先用 localStorage 缓存瞬时渲染 → 再拉后端权威配置覆盖 → 保存/重置后回写缓存。**API Key 永不入缓存**。
- **系统提示词**:已作废前端编辑,完全由后端配置(见下)。

---

## 四、提示词与配置归属

- **系统提示词**:由后端独占管理(代码默认值 `DEFAULTS` 或 `data/config.json`),前端不可见、不可改。模型据此完成意图判断与结构化输出。
- **输出协议**:模型回复必须以模式标记开头(`【模式：生成】` / `【模式：对话】`),后端据此分支:
  - 对话模式 → 直接返回文本,跳过生图。
  - 生成模式 → 提取 `【Prompt】`,调用图像模型,下载落盘,返回文件名。
- **调用稳定性**:润色调用采用较低温度(`temperature 0.3`),提升意图判断与格式的可解析性。

## 五、数据与存储

| 内容 | 位置 | 说明 |
|---|---|---|
| 生成图片 | `data/images/` | 随机文件名,经 `/api/images/:filename` 访问 |
| 会话记录 | `data/sessions/<id>.json` | 每会话一文件,含完整消息 |
| 非敏感配置 | `data/config.json` | baseUrl、模型名等 |
| API Key | `.env`(优先)/ `config.json` | env 优先级最高;GET 接口永不回传明文 |
| 前端配置缓存 | 浏览器 localStorage | 仅非敏感项,断网回退用 |

路径穿越防护:图片读取/删除严格限定在 `images/` 目录内(`path.resolve` 前缀校验)。

---

## 六、安全要点

1. **密钥隔离**:API Key 仅存服务端,前端只拿到 `hasApiKey` 布尔标识。
2. **提示词不外泄**:系统提示词不下发前端,防止被逆向/篡改。
3. **路径安全**:图片操作做目录越界校验;上传/生成图统一随机命名。
4. **上传限制**:请求体大小上限,防超大 base64。
5. **历史遗留**:`demo/image.py` 中曾硬编码真实 API Key,迁移前应已轮换;新密钥只入 `.env`。

## 八、典型使用流程

1. **首次使用**:在「设置」填入 DashScope API Key 并保存(服务端持久化)。
2. **生图**:在「对话」输入"画一双红色网面运动鞋,白色鞋底" → 助手判断为生成 → 回复理由+设计说明并出图。
3. **改图**:上传/复用一张参考图作输入,输入"把鞋帮改成翻毛皮" → 图生图产出新款。
4. **咨询**:输入"这个设计适合运动场景吗?" → 助手纯文字回复建议,不生图。
5. **管理**:在「图库」查看/下载/删除历史作品;在会话面板切换不同对话线。

## 十、迁移自 `autoShoes` 的关键变化

- Electron 主进程 → Node 后端;渲染进程 → Vite 前端。
- 单 IPC 桥 → REST API(`/api/*`,统一信封 `{success,data,error}`)。
- `file://` 图片协议 → `/api/images/:filename` 静态路由。
- 单一历史文件 → 多 session 服务端持久化 + 多轮上下文。
- "逢消息必生图" → 意图驱动(生成 / 对话)。
- 提示词前端可编辑 → 后端独占配置。
- 预览弹窗去重 → 前端共享 `ImageModal`(预览/复制/下载/复用)。
