# 修改策略:从「生成中心」转向「对话为主、生图触发式」

> 现状诊断 → 目标模型 → 架构级改动策略(后端/提示词/前端/数据)→ 触发机制方案 → 分阶段实施 → 取舍。

---

## 一、问题诊断:为什么会"倒置"

当前表面上已有"生成/对话"双模式,但**架构骨架仍是生成中心的**,表现为:

1. **单一入口承担一切**:`POST /api/chat/generate` 是唯一对话端点,**每条消息**都进入"多模态润色 → 模式标记判断 → 可能生图"的管线。即便最终是"对话模式",也披着 generate 的外壳走过同一套重提示词协议。
2. **模型被迫每轮做生图决策**:系统提示词强制每条回复以 `【模式：…】` 开头。这把"是否消耗生图额度"的决定权交给 LLM 的每轮判断——不稳定、易误触、且让产品气质偏向"生图工具"。
3. **UI 全面倾斜生图**:标题"AI 生图"、`生成中…` 徽标、思考文案、会话即"生图会话"。普通咨询也显得是在"等待生图"。
4. **提示词复杂度反噬**:为压住误生图,提示词越写越长越严,反而降低遵循度,出现"该对话却生图""生图却不给理由"。

**根因**:把"是否生图"做成了**每轮自动推断**,而非**用户的显式动作**。只要这个前提不变,怎么调都是生成中心。

---

## 二、目标模型:对话为主、生图为辅(触发式)

- **默认路径 = 对话**:每条消息默认得到一个**自然对话回复**(关于鞋类的设计、建议、说明、闲聊)。这条路径**绝不生图**,也**不要求模型输出模式标记**——它就是一个正常的助手回复。
- **生图 = 显式触发的独立动作**:只有当用户**主动触发**生成时,才进入生图管线(构造 Prompt → 调图像模型 → 落盘)。
- **两条路径共享同一会话上下文**:对话积累的设计共识(款式/配色/材质)会成为生图的输入,使"聊着聊着,点一下就出图"成立。

一句话:**把"生图决策"从模型每轮推断,上移为用户的一次显式点击。**

---

## 三、核心策略:架构层分离两条调用路径

### 3.1 后端:拆分端点
| 现状 | 目标 | 职责 |
|---|---|---|
| `POST /api/chat/generate`(全能) | **`POST /api/chat/reply`** | 纯对话:多模态助手自然回复,**不解析模式、不生图**。每条消息默认走这里。 |
| 同上 | **`POST /api/chat/generate`**(保留,语义收窄) | 显式生图:仅由用户触发。输入"生图指令 + 会话上下文" → 产出 设计说明+Prompt → 调图像模型 → 落盘 → 返回图片。 |

- `reply` 与 `generate` 都读取同一 session 的历史作为上下文,都把消息写回 session(对话写文字消息,生图写文字+图片消息)。
- `reply` 不再需要 `【模式】` 协议;`generate` 不再需要"判断要不要生图"(既然被调用了,就是要生图),只专注产出高质量 Prompt。

### 3.2 提示词:双提示词
| 路径 | 提示词 | 形态 |
|---|---|---|
| 对话 `reply` | **对话助手提示词** | 自然、简洁的鞋类设计顾问人设。正常答疑/建议/讨论。**允许**在判断用户可能想要图时,用一句话提示"需要的话可以点【生成图片】",但**不自行出图、不输出结构化标记**。 |
| 生图 `generate` | **生图提示词** | 给定"用户生图指令 + 会话上下文",输出 `【理由】【设计说明】【Prompt】`,供图像模型使用。无需再做"要不要生图"的判断。 |

- 提示词仍由**后端独占配置**(沿用上一轮决策),前端不接触。
- 两个提示词在后端各为一份默认值(`config.service` 内 `DEFAULTS` 拆为 `chatPrompt` + `imagePrompt`)。

### 3.3 模型调用环节优化
- 对话路径:可考虑用**更便宜/更快的文本模型**(若多模态非必需),或继续用多模态(支持参考图理解)。`temperature` 可略高(0.5~0.7)让对话自然。
- 生图路径:保持低温度(0.3)确保 Prompt 稳定可解析。
- 由于对话不再触发生图,**绝大多数轮次省下图像调用成本**——这是该策略的直接收益。

## 四、前端 UX 改动策略(对话为主视觉)

| 位置 | 现状 | 目标 |
|---|---|---|
| 页头标题 | "AI 生图" | 改为"鞋类设计助手"等**对话向**称谓;会话面板同理 |
| 输入区 | 单一"发送" | **默认发送=对话回复**;新增显式**【生成图片】按钮**(与发送并列)。点发送走 `reply`,点生成走 `generate` |
| 状态文案 | "生成中…" | 对话时显示"思考中…"/打字指示;仅触发生图时才显示"生成中…" |
| 参考图 | 上传即随消息发 | 上传的参考图在对话轮也可作为视觉上下文传给 `reply`(让助手能"看图讨论"),但**不等于**要生图 |
| 气泡 | 文字+生图 | 不变;生图结果仍以图片气泡呈现,且保留"用作参考图"复用 |
| 空态欢迎 | 引导生图 | 引导**先对话**(描述需求/提问),再视情况触发生图 |

**关键交互原则**:Enter = 对话;「生成图片」按钮 = 显式出图。两者输入框共用,按"用户选择的动作"分流到不同后端路径。

## 五、触发机制方案(需你拍板一处)

> 这是本次大改的**核心决策点**,影响后续所有实现。给出推荐 + 备选。

**方案 A(推荐):输入区显式【生成图片】按钮**
- 输入框旁加一个"生成图片"按钮。用户正常聊天用发送;想出图时点该按钮(可带当前输入文本作为生图指令,无文本则依据上下文派生)。
- 优点:最可预测、零误触、彻底消除"倒置";实现成本低(前端按钮 + 后端复用现有 generate 逻辑)。
- 缺点:多一个按钮,新用户需发现。

**方案 B:助手在对话中"提议生成",用户点提议即触发**
- 对话回复里,当助手判断用户可能想要图,返回一个结构化"可生成"信号;前端渲染成内联芯片/按钮,用户点击才真正生图。
- 优点:更自然、更"对话感";决策仍由用户最终点击(非自动)。
- 缺点:需在对话回复里嵌入轻量结构化信号(弱化版的标记协议),复杂度与误判风险回升。

**方案 C:A + B 混合**
- 默认显式按钮(A);同时允许助手在合适时机提议(B)。最灵活但工作量最大。

> 推荐 **A** 起步,后续按需叠加 B。本策略余下内容按 A 展开。

## 六、数据与类型影响

- `ChatGenerateRequest`:`reply` 与 `generate` 入参可复用(均含 `sessionId / userText / imageBase64`)。可抽一个 `ChatRequest` 共用,或分别定义。
- `ChatGenerateResponse`:`reply` 返回 `{sessionId, content}`(无图片);`generate` 返回 `{sessionId, content, images, count}`。建议:
  - 新增 `ChatReplyResponse { sessionId, content }`。
  - `generate` 响应去掉 `mode` 字段(已无意义,因为 generate 必然是生图)。
- `Session`/消息结构不变(对话消息与生图消息都写入 `messages`,生图消息带 `generatedImages`)。
- 前端 `useChat`:`send`(对话)与 `generate`(生图)两个动作,分别调不同端点。

## 七、分阶段实施(每阶段可独立编译验证)

1. **后端拆路径**:新增 `/api/chat/reply`(纯对话,无模式协议);收窄 `/api/chat/generate`(只做生图,不再判断意图)。`config.service` 拆 `chatPrompt`/`imagePrompt`。
2. **共享类型更新**:加 `ChatReplyResponse`,`generate` 去 `mode`。
3. **前端动作分离**:`ChatInput` 加【生成图片】按钮;`useChat` 拆 `sendReply` / `sendGenerate`;默认发送走 `reply`。
4. **UI 去生成中心化**:标题、状态文案、欢迎语、会话命名改为对话向。
5. **提示词重写**:对话提示词(自然顾问)、生图提示词(专注产出 Prompt)。
6. **校验**:前后端 typecheck + build;真实联调(配 Key)验证"对话不出图、点生成才出图"。

## 八、保持不变的部分(降低改动面)

- 会话机制(session 列表/切换/新建/删除、默认最新、多轮上下文)——两条路径共享。
- 图库、图片静态访问、下载/复制、路径安全。
- 设置页(含 localStorage 缓存策略)、密钥服务端托管、提示词后端独占。
- 图片复用(图→输入)、`ImageModal` 共享组件。
- 存储结构(`data/images`、`data/sessions`)、统一响应信封。

---

## 九、风险与权衡

- **风险**:对话路径若也允许多模态(看参考图讨论),成本仍来自多模态调用(但远低于生图)。可按需切文本模型降本。
- **权衡(方案 A)**:把"是否生图"决定权完全交给用户,牺牲一点"智能感",换来**可预测、可控成本、消除倒置**——与你的诉求一致。
- **过渡**:可保留 `generate` 端点兼容旧调用,前端切换完成即完成迁移,无需数据迁移。

---

## 十、一句话总策略

> **把生图从"每轮自动推断"改为"用户显式触发",对话成为默认路径;后端拆 `reply`/`generate` 双端点,提示词拆对话/生图双份,前端用"发送=对话、生成按钮=出图"的分流交互。**
